## Skeptic Report — design gate (round 1)

### What I verified (with evidence)

1. **Ticket AC vs. design coverage** — Read `ticket.md` (and fetched from Linear) and cross-referenced every AC against `design.md`, `tasks.md`, and the three spec deltas.

2. **Flyway version** — Confirmed the current latest migration in `backend/src/main/resources/db/migration/` is V38. The design correctly targets V39. No version conflict.

3. **`resource_permissions` RLS gap analysis** — Read V36 SQL in full. The four policies on `resource_permissions` (`_select`, `_insert`, `_update`, `_delete`) each JOIN *only* to `dashboards.owner_id`. A pipeline owner running under the app pool (which has RLS active) cannot INSERT a pipeline grant row because `EXISTS (SELECT 1 FROM dashboards d WHERE d.id = resource_permissions.resource_id AND d.owner_id = ...)` will be false for a `resource_type = 'pipeline'` row. The design's Decision D1 and its Risk/Mitigation bullet correctly identify this and plan to add OR-ed pipeline-owner policies in V39. This is sound.

4. **`PipelineStepRepository` owner-JOIN problem (critical gap)** — Read `PipelineStepRepository.scala` in full. Every method (`listByPipeline`, `findById`, `insert`, `update`, `delete`) joins `pipeline_steps` to `pipelines` on `pipeline.ownerId === ownerUuid`. A viewer or editor grantee passes a correct `user` but `ownerUuid` will not match `pipelines.owner_id`. This means:
   - `PipelineService.listSteps` calls `pipelineRepo.exists(pipelineId, user)` (owner-only) then `pipelineStepRepo.listByPipeline(pipelineId, user)` (also owner-only JOIN). After the design change, the `exists` call is replaced with `findByIdShared` in the service, but `pipelineStepRepo.listByPipeline` still does the owner JOIN — a viewer grantee would get an empty step list even with a valid pipeline in hand.
   - `PipelineService.analyze` calls both `pipelineRepo.findSummaryById(pipelineId, user)` (owner-only) AND `pipelineStepRepo.listByPipeline(pipelineId, user)` (owner-only JOIN). The design replaces `findSummaryById` with `findByIdShared` (task 3.1) but says nothing about `analyze`. A viewer calling `/api/pipelines/:id/analyze` would get a `None` summary from the sharing-aware path but an empty step list from the owner-only step repo — producing a confusing analyze response.
   - `PipelineService.updateStep` calls `pipelineStepRepo.findById(stepId, user)` directly — editor grantees will always get `None` from that owner-scoped JOIN.
   - `PipelineService.deleteStep` calls `pipelineStepRepo.delete(stepId, user)` — same issue for editors.
   
   **The design has no task or decision that addresses the owner-scoped JOIN in `PipelineStepRepository`**. The design acknowledges this table exists but only adds `findByIdShared` / `findByIdOwned` / `existsForShared` to `PipelineRepository`. Step-level operations will be silently broken for all non-owner grantees until `PipelineStepRepository` is also updated (or the service layer bypasses its owner predicates via an internal/system-context path).

5. **`analyze` endpoint omitted from task list** — The spec says viewer grantees can call `GET /api/pipelines/:id/analyze` (pipeline-sharing spec, Viewer Requirement). The task list (tasks 3.1–3.5) covers `findSummaryById`, `delete`, `updateName`, `addStep`/`updateStep`/`deleteStep`, and `triggerRun`. There is no task to update `PipelineService.analyze` to use sharing-aware repository calls. This is an uncovered AC path.

6. **`PipelineRunService.pipelineExists` not mentioned in design** — Read `PipelineRunService.scala`. The SSE guard (`PipelineRunStreamRoutes`) calls `runService.pipelineExists(pipelineId, user)`, which delegates to `pipelineRepo.findById(pipelineId, user)` — the owner-only method. Decision D6 says the routes should call `findByIdShared` first, but task 4.1 says to update `PipelineRunStreamRoutes`. The route currently calls through `PipelineRunService.pipelineExists`. The task either (a) needs to update `PipelineRunService.pipelineExists` to be sharing-aware, or (b) bypass the service method entirely in the route. This is a one-liner gap but it is currently unspecified — "update PipelineRunStreamRoutes to use sharing-aware existence check" does not say *how* (which method to call, whether to add a service method).

7. **`findSummaryById` vs. `findById` call in `analyze`** — `PipelineService.analyze` calls both `pipelineRepo.findSummaryById(pipelineId, user)` AND `pipelineRepo.findById(pipelineId, user)`. Task 3.1 updates `findSummaryById` to use the sharing-aware path. `findById` (owner-only) is still called in `analyze`. Design Decision D2 keeps `findById` unchanged. This means `analyze` will succeed the summary lookup (sharing-aware) but fail the pipeline detail lookup (owner-only) for non-owner callers, producing a `case _` NotFound even for valid grantees.

8. **`PipelineRunHistoryRoutes` and `PipelineRunSubmitRoutes` not addressed in tasks** — Read `PipelineRunSubmitRoutes.scala` and `PipelineRunHistoryRoutes.scala` are implied to need changes for viewer/editor access. They are not explicitly listed in tasks 4.x. Task 4.1 only mentions `PipelineRunStreamRoutes`. The spec's "Viewer can read run history" and "Editor can trigger a run" require these routes to be updated. The task list is silent on them.

9. **Spec coverage check** — The pipeline-sharing spec covers all six ticket ACs. The resource-permissions spec covers the modified table semantics. The pipeline-run-sse spec covers the SSE viewer/editor grant path. No gaps in spec coverage itself.

10. **Design D5 claim: "No share dialog exists in the dashboard today"** — This is stated as a fact in Decision D5. The design acknowledges this PR will introduce the first share dialog in the app. This is accurate per the codebase (no `ShareDialog` or `PermissionDialog` component found in `frontend/src`). No problem here.

11. **No schema file for the new `/api/pipelines/:id/permissions` endpoint** — The change adds new REST endpoints. The specs directory contains a `pipeline-sharing/spec.md` but no OpenAPI path delta or JSON schema file for the request/response shapes (`GrantPermissionRequest`, `PermissionResponse`). The dashboard permission endpoint's existing wire shape (in `GrantPermissionRequest`, `PermissionResponse`) will be reused, but there is no spec delta confirming this reuse or documenting the pipeline permission response shape. CLAUDE.md states: "Schemas in `schemas/` (JSON Schema 2020-12) and specs in `openspec/` (OpenAPI) define the contract." If the existing schemas cover this by reuse, that should be stated explicitly in the design.

### Verdict: REFUTE

### Change Requests

1. **[Blocking] `PipelineStepRepository` owner-JOIN is unaddressed.** Every method in `PipelineStepRepository` hard-codes `pipeline.ownerId === ownerUuid` in its JOIN predicate. When an editor or viewer grantee calls step endpoints, these joins silently return no rows regardless of any sharing-aware gate in `PipelineService`. The design must either: (a) add sharing-aware variants to `PipelineStepRepository` (e.g. `listByPipelineShared(id, callerOpt)`, `findByIdShared`, etc.) that drop the owner constraint and use an existence-checked pipeline fetch instead, OR (b) document that the step repo will run under system context after the service has confirmed sharing access. Add tasks for this. Without this fix, AC2 (viewer can list steps) and AC3 (editor can mutate steps) cannot be satisfied.

2. **[Blocking] `PipelineService.analyze` is not updated in the task list.** The `analyze` method calls `pipelineRepo.findSummaryById(pipelineId, user)` (covered by task 3.1) AND `pipelineRepo.findById(pipelineId, user)` (owner-only, not covered) AND `pipelineStepRepo.listByPipeline(pipelineId, user)` (owner-only, not covered). A viewer calling `GET /api/pipelines/:id/analyze` will get a consistent 404 from the owner-only `findById` call regardless of the sharing-aware summary path. Add a task to update `analyze` to use `findByIdShared` (or the internal method with a sharing pre-check), and to use a sharing-aware step list fetch.

3. **[Blocking] `PipelineRunHistoryRoutes` and `PipelineRunSubmitRoutes` missing from task 4.** The spec mandates viewer access to run history (`GET /api/pipelines/:id/run-history`) and editor access to trigger runs (`POST /api/pipelines/:id/run`). These routes need their existence/ACL checks updated for sharing. They are not mentioned in tasks 4.x. Add explicit tasks for each.

4. **[Blocking] `PipelineRunService.pipelineExists` is owner-only and task 4.1 does not specify how to fix it.** The SSE route calls `runService.pipelineExists(pipelineId, user)` which delegates to `pipelineRepo.findById` (owner-only). Task 4.1 says "update `PipelineRunStreamRoutes` to use sharing-aware existence check" but is silent on whether this requires a new `PipelineRunService` method or a direct repo call. The design decision D6 also says "resolve `findByIdShared(id, callerOpt)` first" but `PipelineRunStreamRoutes` takes `user: AuthenticatedUser`, not `Option[AuthenticatedUser]`. Specify concretely: either add `pipelineExistsShared(id, user): Future[Boolean]` to `PipelineRunService` (calling `findByIdShared`), or update task 4.1 to reflect a bypass through a service method. The implementer will otherwise make an ambiguous call.

5. **[Non-blocking] Confirm OpenAPI/schema reuse for permission endpoints.** The design introduces `/api/pipelines/:id/permissions` with `GET`, `POST`, `DELETE` sub-routes. The existing `GrantPermissionRequest` and `PermissionResponse` types (used by the dashboard permission routes) will presumably be reused. The spec deltas do not confirm this or provide an OpenAPI path entry for the new routes. Either add an OpenAPI delta to `specs/` or add a sentence in `design.md` stating these wire types are deliberately reused unchanged.

### Non-blocking notes

- Decision D3 says "Owner or Editor: allow mutation" but does not say how `ResourceAccess` is threaded. `AccessChecker.requireAccess` returns `Either[ServiceError, ResourceAccess]`. The service will need to call this (or `findByIdShared`) and branch on the returned `ResourceAccess` level. The tasks assume this is understood from the dashboard pattern — acceptable, but worth noting that `requireAccess` is the right call site since `requireOwnerOnly` will reject editors.
- The test task 7.2 ("extend `PipelineAclSpec` with one regression test") is well-scoped. Ensure the regression test asserts a concrete route (e.g. `GET /api/pipelines/:id`) returns 404 for a cross-user no-grant caller after the sharing change is applied, to prevent future regressions from silently opening the owner-only path.
