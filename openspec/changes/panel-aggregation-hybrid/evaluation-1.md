## Evaluation Report — Cycle 1

### Phase 1: Spec Review — FAIL
Issues:
- **Critical**: The `panel-viz-aggregation` spec's core requirements ("Metric renders avg aggregate over all rows", "apply_proposal persists the aggregation spec on the created panel") are not durably satisfied. The `aggregation` field is computed correctly in-memory (decode/patch/applyPatch all correct), but there is no database persistence path for it at all — see Phase 2 for the root cause. The very first PATCH-then-reload of a metric's aggregation spec silently drops it (reproduced live in Phase 3). This is not a partial edge case; it is the primary path through the feature.
- Planning-artifact gap: neither `design.md` (Decision 5) nor `tasks.md` (section 1) calls out the actual DB schema/repository wiring needed to durably store `aggregation` (a new column + migration, or folding it into the existing `field_mapping` JSONB blob, plus updating `PanelRepository`/`PanelRowMapper`). Design.md Decision 5 says the field is "stored/passed through opaquely as JSON" but never identifies *where* — the plan is incomplete, not just the implementation. This needs to be corrected in design.md/tasks.md as part of the fix, not just the code.
- All 23 tasks.md items are checked off and match what was literally coded, but task 1.1/1.2 ("bump `format`... `applyPatch`") never included the repository/mapper/migration layer, so "done" here doesn't mean "working end-to-end."
- No scope creep observed; MCP/schema/contract wiring (tasks 2–3) is correct and matches design.

### Phase 2: Code Review — FAIL
Issues:
1. **Critical / missing persistence path**: `backend/src/main/scala/com/helio/infrastructure/PanelRowMapper.scala:86-96` (`metricConfig`/`chartConfig`) never reads an `aggregation` value from the DB row, and `domainToRow` (`PanelRowMapper.scala:51-82`) never writes `mp.config.aggregation`/`cp.config.aggregation` anywhere. There is no `aggregation` column on the `panels` table (confirmed via `\d panels` — only `field_mapping` exists) and no Flyway migration adds one (`backend/src/main/resources/db/migration/` has nothing referencing "aggregation"). `PanelRepository.replace` (`PanelRepository.scala:196-207`), the method `PanelPatchApplier` calls after a config PATCH, explicitly whitelists the columns it persists (`typeId, fieldMapping, content, imageUrl, imageFit, dividerOrientation, dividerWeight, dividerColor, lastUpdated`) — `aggregation` is not among them and there is no column to put it in. Net effect: `MetricPanelConfig.applyPatch`/`ChartPanelConfig.applyPatch` correctly set `aggregation` in memory, but the very next line (`panelRepo.replace(updated, now)`) silently drops it, and the re-read (`rowToDomain` → `metricConfig`/`chartConfig`) returns `aggregation = None` regardless of what was patched. This affects both the `PATCH /api/panels/:id` binding-editor path and any panel re-fetch (`listByDashboard`) after `apply_proposal` creates it — `apply_proposal`'s response looks correct only because `PanelRepository.insert` returns the passed-in in-memory `panel` object rather than a DB re-read (`PanelRepository.scala:154-156`), masking the same gap.
2. Backend unit tests give false confidence: `PanelSpec.scala`'s new `MetricPanelConfig.aggregation`/`ChartPanelConfig.aggregation` tests (`backend/src/test/scala/com/helio/domain/PanelSpec.scala:165-268`) exercise `decode`/`Patch.decode`/`applyPatch` as pure functions only — never through `PanelRowMapper`/`PanelRepository`, so they cannot catch the DB round-trip loss. `DashboardApplyProposalSpec`'s two new aggregation scenarios (lines ~204-247) only assert on the immediate HTTP response from `apply_proposal`, which (per point 1) is built from the in-memory object, not a re-read from Postgres — so those tests pass despite the data never actually landing durably. No backend test does "PATCH with aggregation → re-fetch (`GET`/`listByDashboard`) → assert aggregation survived," which is the one test that would have caught this.
3. Minor / non-blocking: `frontend/src/features/panels/state/panelThunks.ts:169-180` branches between a 4-arg and 5-arg call to `updatePanelBindingRequest` "to preserve the exact 4-arg call shape," but calling a function with an explicit `undefined` 5th argument is behaviorally identical to omitting it in JS/TS — the ternary is dead complexity that doesn't change behavior. Simplify to a single call passing `aggregation` unconditionally.
4. Minor / non-blocking: no frontend test asserts the aggregation-aware page-size override in `usePanelData.ts:88-98` (`pageSize = Number.MAX_SAFE_INTEGER` when an aggregation spec is present) actually takes effect at a row count that would matter — all `usePanelData.test.ts` aggregation fixtures use ≤3 rows, well under the pre-existing metric/chart defaults (10/200), so a regression reverting the page-size override wouldn't be caught by unit tests. (Not independently blocking given the critical issue above, but worth closing in the same pass.)

Everything else reviewed cleanly: `CONTRIBUTING.md` mechanical rules (no inline FQNs, `check:scala-quality` clean, no new file-size violations), `DESIGN.md` token usage in `BindingEditor.tsx`'s new Aggregation sub-section (reuses existing `Select`/`panel-detail-modal__*` classes, no ad-hoc styling), frontend `computeAggregate`/`groupAndAggregate` semantics (correctly mirrors `AggregateStep`, including the empty-string-vs-null coercion guard flagged in the round-2 design review), MCP/schema contract wiring, and the absent-vs-null PATCH shape convention.

### Overall: FAIL

### Change Requests
1. Add a durable persistence path for `MetricPanelConfig.aggregation` / `ChartPanelConfig.aggregation`: either (a) a new Flyway migration adding an `aggregation` JSONB column to `panels` plus reading/writing it in `PanelRowMapper.metricConfig`/`chartConfig`/`domainToRow`, and adding it to `PanelRepository.replace`'s persisted-column list (`PanelRepository.scala:196-207`); or (b) fold `aggregation` into the existing `field_mapping` JSONB blob at the mapper boundary (documenting why, since the domain model keeps them as separate fields). Whichever approach, `PanelRepository.insert`/`replace`/`rowToDomain` must round-trip `aggregation` losslessly.
2. Add at least one backend test per panel type that goes through the real repository round trip: `PATCH /api/panels/:id` with an aggregation spec → `GET`/`listByDashboard` (a fresh read, not the PATCH response) → assert `config.aggregation` survived. The existing `PanelSpec.scala`/`DashboardApplyProposalSpec` tests do not cover this and must remain in addition, not be replaced.
3. Update `design.md` Decision 5 and `tasks.md` section 1 to explicitly name the DB/repository wiring task (this was the actual gap — the plan was incomplete, not just the code).
4. Re-run the Phase 3 manual flow (metric aggregation save → reload/reopen, and the equivalent chart `groupBy` flow) after the fix and confirm the aggregate renders and survives a page reload, not just the same session.

### Non-blocking Suggestions
- `frontend/src/features/panels/state/panelThunks.ts:169-180` — simplify the 4-arg/5-arg conditional call to `updatePanelBindingRequest` to a single unconditional call; passing `aggregation: undefined` explicitly is behaviorally identical to omitting the argument.
- Add a `usePanelData.test.ts` case with a row count exceeding the pre-existing metric (10) / chart (200) default page size to actually exercise the `pageSize = Number.MAX_SAFE_INTEGER` override in `usePanelData.ts:88-98`; current fixtures (≤3 rows) don't distinguish the override from the old default.
