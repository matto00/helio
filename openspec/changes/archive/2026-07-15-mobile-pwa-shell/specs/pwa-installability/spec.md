## ADDED Requirements

### Requirement: Web app manifest
The frontend build SHALL emit a web app manifest with `name: "Helio"`, `short_name: "Helio"`,
`display: "standalone"`, `start_url: "/"`, `scope: "/"`, and `background_color`/`theme_color` matching the
light-theme `--app-bg` value from `theme.css`. The manifest SHALL reference icons generated from
`public/orbit-mark.svg`: 192×192 and 512×512 standard icons and a 512×512 maskable icon.

#### Scenario: Manifest is served and linked
- **WHEN** the production build is loaded in a browser
- **THEN** `index.html` links a manifest whose `display` is `standalone`, whose `scope` and `start_url` are `/`,
  and whose icon set includes 192, 512, and 512-maskable entries

### Requirement: iOS installability meta tags
`index.html` SHALL include `viewport-fit=cover` in the viewport meta, `apple-mobile-web-app-capable`,
`apple-mobile-web-app-status-bar-style`, an `apple-touch-icon` link to a 180×180 icon, and two
`<meta name="theme-color">` tags gated on `prefers-color-scheme` sourcing the light and dark `--app-bg` values.

#### Scenario: iOS meta tags present
- **WHEN** `index.html` is inspected
- **THEN** it contains `viewport-fit=cover`, `apple-mobile-web-app-capable`, an `apple-touch-icon` link, and
  dual `theme-color` metas for light (`#f4f2ed`) and dark (`#121110`)

### Requirement: App-shell precache with auto-update
The frontend SHALL register a service worker generated by `vite-plugin-pwa` (Workbox `generateSW`) with
`registerType: "autoUpdate"`. The precache manifest SHALL cover only built app-shell assets (JS, CSS, HTML,
icons). The service worker SHALL NOT be enabled in dev mode.

#### Scenario: Service worker registers in production build
- **WHEN** the production build is served and loaded
- **THEN** a service worker registers and precaches the built JS/CSS/HTML/icon assets

#### Scenario: New deploys replace the service worker automatically
- **WHEN** a new build is deployed and a client with the old service worker revisits
- **THEN** the new service worker activates without requiring a manual user action

### Requirement: API responses are never cached
The service worker SHALL treat everything under `/api/**` as network-only: no runtime caching rule may match
`/api`, and the SPA navigation fallback SHALL exclude `/api/**` via `navigateFallbackDenylist`. Authenticated
API responses SHALL never be served from any service-worker cache, preserving the semantics of the global 401
interceptor in `main.tsx`.

#### Scenario: API requests bypass the cache
- **WHEN** the app requests any `/api/**` URL while the service worker is active
- **THEN** the response comes from the network, and no `/api/**` response is stored in or served from a cache

#### Scenario: OAuth navigation is not intercepted by the SPA fallback
- **WHEN** the browser performs a full navigation to `/api/auth/google` on the same origin (dev/preview proxy)
- **THEN** the service worker does not serve `index.html` for it; the navigation reaches the server

#### Scenario: 401 handling is preserved
- **WHEN** an authenticated API call returns `401 Unauthorized` while the service worker is active
- **THEN** the 401 reaches the axios interceptor and the existing clear-auth/redirect behavior runs unchanged

### Requirement: Fonts are cache-first
The service worker SHALL cache the Google Fonts trio (Fraunces, Schibsted Grotesk, JetBrains Mono):
stale-while-revalidate for `fonts.googleapis.com` stylesheets and cache-first with a long expiry for
`fonts.gstatic.com` font files. The font trio SHALL NOT be removed from `index.html`.

#### Scenario: Standalone cold load uses cached fonts
- **WHEN** the installed app cold-launches after fonts were cached on a prior visit
- **THEN** font files are served from the service-worker cache and text renders without a flash of unstyled text
