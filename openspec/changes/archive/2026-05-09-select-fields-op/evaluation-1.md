## Evaluation Report — Cycle 1

### Phase 1: Spec Review — FAIL

Issues:

- **[FAIL] AC: "The frontend step config UI renders a checklist of available fields"** — The checklist
  can never be populated in the current implementation. `runPipeline` in `pipelineService.ts` is typed
  as `Promise<{ runId: string }>`, but the backend's `POST /api/pipelines/:id/run` returns
  `{ rowCount, rows }` synchronously — there is no `runId` in the response. Because
  `action.payload.runId` is always `undefined`, `state.runId` is set to `undefined`, the polling
  `useEffect`'s guard (`!runId`) short-circuits the interval, `setRunStatus` is never called with
  rows, and `runResult` stays `null` forever. Confirmed by live browser test and direct curl:
  ```
  POST /api/pipelines/:id/run → {"rowCount":5,"rows":[...]}  (no runId field)
  GET  /api/pipelines/:id/runs/:runId → 404  (endpoint exists but returns "Run not found")
  ```
  After clicking "Run pipeline ▶", the status badge stays on "Queued…" indefinitely; no polling
  requests are made; the step card always shows the prompt, never the checklist.

- **[FAIL] Task 2.4 — "Wire checkbox toggle to update the step's config `fields` array via
  `PATCH /api/pipeline-steps/:id`"** — `handleFieldToggle` calls
  `updatePipelineStep(step.id, ...)` but `step.id` is always a local counter ID (`step-1`,
  `step-2`, …) generated by `makeStep()`. No backend step with that ID exists, so every PATCH
  silently 404s. The code's own comment acknowledges the problem
  (`// The step.id may be a local ID (step-N)`). Task 2.4 is marked `[x]` but config persistence
  never actually occurs.

- **[FAIL] `selectedFields` never hydrated from persisted config** — `useState<string[]>([])` always
  starts empty. The local `Step` interface (`{ id, opType, label, rowCount }`) carries no `config`
  field, so even if PATCH did succeed, reopening the card would show all boxes unchecked.

- [PASS] Flyway V25 migration correctly drops and re-adds `pipeline_steps_op_check` including `'select'`.
- [PASS] `allowedOps` in `PipelineStepRoutes` updated; route test added.
- [PASS] `applySelect` implemented and wired into `applyStep`; three engine tests cover subset,
  missing field, and empty list.
- [PASS] `OP_TYPES` updated with `{ id: "select", label: "Select fields", icon: "☑" }`.
- [PASS] `SelectFieldsConfig` component renders correctly in isolation (prompt / checklist branches).
- [PASS] OpenSpec artifacts (specs, proposal, design, tasks) reflect the implemented design intent.
- [PASS] No scope creep; no regressions to other ops.

### Phase 2: Code Review — FAIL

Issues:

- **[FAIL] Type/runtime mismatch in `runPipeline`** (`pipelineService.ts:53–56`) — Return type is
  `Promise<{ runId: string }>` but the backend returns `{ rowCount: number; rows: [...] }`.
  `response.data.runId` is always `undefined`. This is the root cause of `runResult` never being
  populated and the checklist never appearing.

- **[FAIL] `clearRunState` does not reset `runResult`** (`pipelinesSlice.ts:153–157`) — `clearRunState`
  clears `runId`, `runStatus`, and `runError` but not `runResult`. `PipelineDetailPage` dispatches
  `clearRunState` on unmount, so navigating from Pipeline A (which had a successful run and set
  `runResult`) to Pipeline B will display Pipeline A's columns in the select checklist.
  Fix: add `state.runResult = null` to `clearRunState`.

- **[WARN] Silent PATCH always 404s** (`PipelineDetailPage.tsx:162–171`) — The `.catch(() => {})`
  swallows what are structural 404 failures (wrong ID type), not transient network hiccups. While
  optimistic-UI suppression is acceptable in principle, here it masks a completely non-functional
  feature. The comment should at minimum document the known limitation clearly; the guard in CR-4
  below is the proper fix.

- [PASS] `applySelect` is idiomatic Scala: `row.view.filterKeys(fieldSet.contains).toMap` —
  consistent with other ops, silent-omission behavior documented in D2.
- [PASS] No `any` types introduced; `Record<string, unknown>[]` is appropriate.
- [PASS] `SelectFieldsConfig` is clean and composable; props are well-typed.
- [PASS] Backend engine tests are meaningful and cover all three required scenarios.
- [PASS] `SelectFieldsConfig` unit tests cover checklist render, prompt render, and both toggle

### Change Requests

1. **Fix `runPipeline` service to match the backend's synchronous response**
   (`frontend/src/services/pipelineService.ts:53–56`):
   Change the function's return type from `Promise<{ runId: string }>` to
   `Promise<{ rowCount: number; rows: Record<string, unknown>[] }>` and update the
   `httpClient.post<...>` type argument to match:
   ```ts
   export async function runPipeline(
     pipelineId: string,
   ): Promise<{ rowCount: number; rows: Record<string, unknown>[] }> {
     const response = await httpClient.post<{ rowCount: number; rows: Record<string, unknown>[] }>(
       `/api/pipelines/${pipelineId}/run`,
     );
     return response.data;
   }
   ```

2. **Update `submitPipelineRun` thunk and slice to handle synchronous run result**
   (`frontend/src/features/pipelines/pipelinesSlice.ts`):
   - Change the thunk's generic return type from `{ runId: string }` to
     `{ rowCount: number; rows: Record<string, unknown>[] }`.
   - In the `fulfilled` handler, replace the current body with:
     ```ts
     state.runId = null;
     state.runStatus = "succeeded";
     state.runResult = action.payload.rows;
     ```
   - Remove the polling `useEffect` from `PipelineDetailPage.tsx` (lines 476–495) — the run
     result is now available synchronously via the dispatched action, so polling is unnecessary.
   - Update `PipelineDetailPage.tsx` to dispatch `submitPipelineRun` with `.unwrap()` in
     `handleRunPipeline` and call `dispatch(fetchPipelineRunHistory(id))` on success, replacing
     the polling interval.

3. **Add `state.runResult = null` to `clearRunState`**
   (`frontend/src/features/pipelines/pipelinesSlice.ts:153–157`):
   ```ts
   clearRunState(state) {
     state.runId = null;
     state.runStatus = null;
     state.runError = null;
     state.runResult = null;   // add this line
   },
   ```

4. **Guard `updatePipelineStep` PATCH against local placeholder IDs**
   (`frontend/src/components/PipelineDetailPage.tsx:162–171`):
   Local step IDs (`step-N`) never correspond to backend records. The PATCH always fails silently.
   Suppress the call for local IDs:
   ```tsx
   function handleFieldToggle(field: string, checked: boolean) {
     const next = checked ? [...selectedFields, field] : selectedFields.filter((f) => f !== field);
     setSelectedFields(next);
     // Only PATCH steps that have been persisted (real UUIDs, not local placeholders).
     if (!step.id.startsWith("step-")) {
       void updatePipelineStep(step.id, JSON.stringify({ fields: next })).catch(() => {});
     }
   }
   ```
   The longer-term fix is to POST new steps to the backend when they are added (obtaining a real
   ID), then use that ID for PATCH. That is a broader architectural change; the guard above is the
   minimal correctness fix for this PR.

5. **Hydrate `selectedFields` from the step's persisted `config` on mount**
   (`frontend/src/components/PipelineDetailPage.tsx:160`):
   - Add `config?: string` to the local `Step` interface (line 40–45).
   - Populate `config` in `makeStep` (default `"{}"`) and in the Redux-step-to-local-step mapping
     when `fetchPipelineSteps` results are loaded.
   - Initialize `selectedFields` lazily from config:
     ```tsx
     const [selectedFields, setSelectedFields] = useState<string[]>(() => {
       if (step.opType.id !== "select") return [];
       try {
         return (JSON.parse(step.config ?? "{}") as { fields?: string[] }).fields ?? [];
       } catch {
         return [];
       }
     });
     ```

### Non-blocking Suggestions

- Add `aria-label="Available fields to select"` to the `<ul role="list">` in `SelectFieldsConfig`
  for richer screen-reader context.
- Consider resetting `selectedFields` in a `useEffect` when `step.id` changes to avoid stale
  state if the same `StepCard` component instance is reused for a different step.
- `OP_TYPES` lists `{ id: "aggregate" }` but the backend `allowedOps` uses `"groupby"`. This is a
  pre-existing mismatch (not introduced by this PR) worth fixing in a follow-up.
