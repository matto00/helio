## Context

`user_sessions.token` (V7, completed by V11) stores the raw 64-char hex session
token generated by `AuthService.generateSessionToken()`. Three call sites touch
it directly:

- `UserRepository.createSession` — inserts the row (`AuthService.register` /
  `.login` / `.completeOAuth`).
- `UserRepository.findSession` / `.deleteSession` — used only by
  `AuthService.logout`.
- `SlickUserSessionRepository.findValidSession` — the hot authentication path,
  called from `AuthDirectives.resolveBearer` on every protected request.

`api_tokens` (V42, HEL-148 Phase 1) already does this correctly:
`ApiTokenService.create` generates the raw `helio_pat_<hex>` value, stores only
`sha256Hex(raw)` in `token_hash`, and returns the raw value exactly once in
`CreateApiTokenResponse`. `AuthDirectives.resolveApiToken` hashes the incoming
Bearer value before comparing. This change gives sessions the same property.

## Goals / Non-Goals

**Goals:**

- `user_sessions` stores only `SHA-256(raw token)`, never the raw token.
- Single hashing implementation shared by both credential kinds.
- No change to the `Authorization: Bearer <token>` contract or any response
  shape (`AuthResponse.token` still carries the raw value).

**Non-Goals:**

- Changing token length/entropy, TTL, or generation (`generateSessionToken`
  unchanged).
- Rehashing existing raw rows in place so users stay logged in (see
  Decisions — rejected).
- Touching `api_tokens` itself; it is the reference, not a target.

## Decisions

**Hash at the repository boundary, not in the domain model.** `UserSession`
keeps `token` meaning "the raw token" everywhere in the service layer
(`AuthService`), unlike `ApiToken`, which already models `tokenHash` as a
first-class domain field. Hashing only happens inside `UserRepository`
(`createSession`/`findSession`/`deleteSession`) and
`SlickUserSessionRepository.findValidSession`, right before/after the SQL
call. *Alternative considered:* mirror `ApiToken` exactly (domain model holds
`tokenHash`, raw value threaded separately for the response). Rejected —
`UserSession.token` already flows unchanged into `AuthResponse.token` in three
places (register/login/OAuth); re-plumbing a parallel raw-value channel
through `AuthService` is a bigger diff for no behavioral gain, since the
ticket is explicit this must stay behavior-preserving at the auth contract.

**Extract `TokenHashing.sha256Hex` into `com.helio.infrastructure`.**
`ApiTokenService.sha256Hex` (services layer) becomes a one-line delegate to
the new helper so `AuthDirectives` (which already calls
`ApiTokenService.sha256Hex`) needs no change. *Alternative considered:* leave
`ApiTokenService.sha256Hex` as the single implementation and have
`UserRepository` import it. Rejected — `com.helio.infrastructure` (repositories)
importing from `com.helio.services` inverts the existing dependency direction
(services → infrastructure everywhere else in the codebase, e.g.
`ApiTokenService` → `ApiTokenRepository`).

**Rename `user_sessions.token` → `token_hash`.** Matches `api_tokens.token_hash`
exactly (the parity the ticket title asks for) and makes it self-documenting
that the column is not the bearer value. Constraint
`user_sessions_token_unique` is renamed alongside it for the same reason.

**Migration invalidates rather than rehashes existing rows.** A raw value
already in the column *could* be forward-hashed in place
(`UPDATE ... SET token_hash = encode(digest(token,'sha256'),'hex')` via
pgcrypto) without forcing re-login, since the raw value is still readable at
migration time. This was considered and **rejected** in favor of the ticket's
explicit scope ("expire/invalidate them on rollout... acceptable for a
session store"): it avoids a hash-parity dependency on pgcrypto matching
Java's `MessageDigest` byte-for-byte, and treats a rollout of a security fix
for "tokens are readable by anyone with DB access" as itself a trust boundary
worth resetting. The migration issues a `DELETE FROM user_sessions` before the
rename.

## Risks / Trade-offs

- **[Risk]** Every logged-in user is force-logged-out on deploy. → Mitigation:
  documented in the proposal and PR description; acceptable per ticket scope;
  frontend already redirects to `/login` on a 401 from `/api/auth/me`
  (`frontend-auth-state` capability) — no separate frontend change needed.
- **[Risk]** A future call site queries `user_sessions.token` directly (raw
  SQL, ops script) and silently breaks after the rename. → Mitigation: grep
  confirmed the only references are the three Scala call sites and migrations
  above; no ops scripts or frontend code touch the column.
- **[Risk]** Logout (`findSession`/`deleteSession`) double-hashes if a caller
  passes an already-hashed value. → Mitigation: both methods take the raw
  Bearer token straight from `AuthDirectives`/`AuthRoutes`, same as today;
  no caller ever holds a hash.

## Migration Plan

1. `V45__hash_session_tokens.sql`: `DELETE FROM user_sessions;` then
   `ALTER TABLE user_sessions RENAME COLUMN token TO token_hash;` and
   `ALTER TABLE user_sessions RENAME CONSTRAINT user_sessions_token_unique TO
   user_sessions_token_hash_unique;`.
2. Ship `TokenHashing`, update the three call sites, update
   `session-persistence` spec delta, in the same change (schema + code
   together per repo convention).
3. Rollback: re-running is not idempotent by nature of `DELETE`; a rollback
   would be a new down-migration restoring the raw `token` column name (data
   already lost) — acceptable, matches other irreversible Flyway migrations
   in this codebase (e.g. V40's recursion fix).

## Planner Notes

- Self-approved: column rename (`token` → `token_hash`) and constraint rename,
  as a direct-parity implementation of the ticket's explicit "reach parity
  with API tokens" goal — not a new architectural direction.
- Self-approved: extracting `TokenHashing` rather than leaving hashing
  duplicated — small refactor, no behavior change, keeps the dependency graph
  consistent with the rest of the codebase.
